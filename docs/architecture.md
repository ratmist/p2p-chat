# Архитектура решения

## 1. Общая идея

HexMesh использует гибридную архитектуру:
- signaling server нужен для discovery, offer/answer/ICE и relay/fallback;
- полезная нагрузка, где возможно, уходит по прямому WebRTC DataChannel;
- media stream для звонков также идет по WebRTC;
- при невозможности прямого канала используется relay и/или мультихоп.

## 2. Компоненты

### Клиент
Основные модули:
- `MeshClient.ts` — сетевое ядро;
- `crypto.ts` — криптография и подпись;
- `MeshContext.tsx` — bridge между UI и сетевым ядром;
- React screens — discovery, chat, calls, files, settings.

### Сервер
`server/index.js` реализует:
- WebSocket signaling;
- health endpoint;
- relay приложенческих пакетов;
- rate limiting;
- heartbeat;
- federated local-server bridging.

## 3. Жизненный цикл узла

1. Клиент генерирует `nodeId`.
2. Генерирует Ed25519 key pair.
3. Открывает WebSocket к signaling server.
4. Отправляет `REGISTER`.
5. Получает `REGISTERED` и затем `NODE_LIST`.
6. Для прямой сессии запускает WebRTC offer/answer/ICE.
7. После открытия DataChannel отправляет пакеты напрямую.

## 4. Топология

Поддерживаются сценарии:
- star через signaling server;
- direct P2P между соседями;
- частичная mesh-сеть из нескольких узлов;
- federated LAN из нескольких signaling серверов.

## 5. Протокол сообщений

Структура `HexPacket`:
- `id` — уникальный идентификатор пакета;
- `type` — тип пакета;
- `from`, `to` — адресация;
- `ttl` — ограничение числа прыжков;
- `visited[]` — список узлов, через которые уже проходил пакет;
- `payload` — данные;
- `sig` — цифровая подпись;
- `ts` — время создания.

## 6. Мультихоп и защита от петель

Алгоритм:
1. Если пакет адресован нам — обрабатываем локально.
2. Иначе уменьшаем `ttl`.
3. Добавляем текущий узел в `visited`.
4. Пытаемся отправить напрямую адресату.
5. Если нельзя — рассылаем соседям, которых нет в `visited`.
6. Если прямых соседей нет — уходим в relay через signaling.

Это дает:
- ограничение времени жизни пакета;
- защиту от бесконечных циклов;
- базовую маршрутизацию без полноценной таблицы маршрутов.

## 7. Надёжность

### Сообщения
- отправка `MSG`;
- постановка в очередь;
- ожидание `ACK`;
- exponential backoff retry.

### Файлы
- `FILE_META` для описания передачи;
- разбиение на чанки;
- `FILE_ACK` на каждый chunk;
- retry для не подтвержденных чанков;
- локальная сверка `SHA-256` после сборки файла;
- pause/resume.

## 8. Звонки

Звонки реализуются через WebRTC:
- signal plane: WebSocket;
- media plane: RTP/RTCP внутри WebRTC stack;
- буферизация и адаптация зависят от браузера / платформы.

Для защиты на демонстрации нужно показать:
- время установления звонка;
- субъективную устойчивость при потере пакетов;
- UI-реакцию на `peer:state`, `call:state`, RTT.

## 9. Поведение при сбоях

### Если падает DataChannel
- удаляется peer;
- звонок переводится в `ended`;
- дальнейшая доставка уходит в relay/fallback.

### Если падает signaling
- запускается reconnect с exponential backoff;
- discovery временно перестает обновляться;
- уже существующий P2P-канал может продолжать работать.

### Если отключается ретранслятор
- мультихоп-маршрут распадается;
- сообщение либо ищет иной маршрут, либо истекает по TTL, либо повторяется позже.

## 10. Сильные стороны архитектуры

- Нет жесткой зависимости от центра для data plane.
- Есть fallback через relay.
- Есть база для локального mesh и federation.
- Протокол уже содержит важные поля для надежности и безопасности.

## 11. Ограничения текущей версии

- Нет полноценной таблицы маршрутов и выбора оптимального next hop.
- Нет устойчивого persisted-store для offline delivery.
- E2E-шифрование payload подготовлено примитивами, но не доведено до end-to-end потока для всех типов данных.
- Нет отдельного congestion control поверх file-transfer protocol, кроме базового throttling.
