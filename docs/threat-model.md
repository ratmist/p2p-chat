# Модель угроз

## 1. Активы, которые нужно защищать
- содержимое сообщений;
- содержимое файлов;
- голосовой трафик;
- идентичность узла;
- доступность сети и приложения;
- корректность маршрутизации;
- целостность передаваемых данных.

## 2. Нарушители

### Нарушитель A: посторонний узел в той же локальной сети
Возможности:
- сканировать сеть;
- пытаться отправлять поддельные пакеты;
- спамить сервер сигналинга;
- пытаться встроиться в relay.

### Нарушитель B: злонамеренный участник mesh
Возможности:
- ретранслировать мусорный трафик;
- создавать петли;
- не отправлять ACK;
- пытаться выдавать себя за другой узел.

### Нарушитель C: пассивный сетевой наблюдатель
Возможности:
- анализировать метаданные;
- перехватывать незашифрованный сигналинг;
- фиксировать частоту, размер и направление трафика.

### Нарушитель D: узел-ретранслятор под контролем атакующего
Возможности:
- выборочно дропать пакеты;
- задерживать пакеты;
- пытаться переупорядочивать пересылку;
- создавать перегрузку канала.

## 3. Основные угрозы

### 3.1 Подмена узла
Риск:
атакующий выдает себя за доверенный узел.

Контрмеры:
- уникальный `nodeId`;
- публичный ключ узла;
- цифровая подпись пакетов Ed25519;
- pinning public key в будущем.

### 3.2 Спам / flood / exhaustion attack
Риск:
атакующий засыпает signaling server и relay-путь пакетами.

Контрмеры:
- rate limiting на сервере;
- ограничение размера пакета;
- ограничение числа параллельных file transfer;
- TTL и `visited[]`;
- backpressure / throttling на передачу файлов.

### 3.3 Петли маршрутизации
Риск:
пакет бесконечно ходит по кругу и забивает сеть.

Контрмеры:
- `ttl`;
- `visited[]`;
- дедупликация по `packet.id` на сервере;
- рекомендуется добавить `seenPackets` на клиенте.

### 3.4 Нарушение целостности файла
Риск:
получатель соберет поврежденный файл.

Контрмеры:
- `SHA-256` по файлу;
- `FILE_ACK` и повторная отправка чанков;
- fail state при несовпадении хэша.

### 3.5 MITM на сигналинге
Риск:
перехват и модификация offer/answer/ICE или relay-пакетов.

Контрмеры:
- WSS/TLS;
- подпись пакетов;
- проверка открытых ключей узлов;
- желательно certificate pinning для production.

### 3.6 Replay attack
Риск:
повторная отправка старого пакета.

Контрмеры:
- `packet.id`;
- `ts`;
- серверная дедупликация;
- желательно ограничение окна допустимого времени и клиентская dedupe-cache.

### 3.7 Анализ метаданных
Риск:
даже при шифровании видно кто с кем и когда общался.

Контрмеры:
- минимизация relay;
- переход на E2E payload encryption для всех пользовательских данных;
- в будущем: padding, cover traffic, batch-forwarding.

### 3.8 DoS через ретранслятор
Риск:
relay-узел может задерживать или дропать пакеты.

Контрмеры:
- retry;
- альтернативный next hop;
- TTL;
- event/logging и health metrics.

## 4. Что защищено сейчас
- базовая идентификация узлов;
- подпись пакетов;
- rate limiting;
- защита от петель на уровне протокола;
- контроль целостности файла;
- базовая устойчивость к отказам.

## 5. Что остаётся уязвимым или частично закрыто
- метаданные общения;
- полное E2E-шифрование для всех типов payload;
- злонамеренный, но валидный relay-узел;
- offline persistence / recovery после долгих разрывов;
- более строгая защита от replay.

## 6. Рекомендуемые доработки
1. Добавить обязательное E2E-шифрование payload для сообщений и file meta.
2. Добавить client-side cache `seenPackets` с TTL.
3. Ввести key pinning и доверенную first-contact модель.
4. Добавить blacklist / penalty для flood-узлов.
5. Добавить формальную политику ротации ключей и восстановление доверия.

## 7. Краткий вывод для защиты

Система защищается от:
- случайных петель;
- базового спама;
- подмены содержимого пакета без ключа;
- повреждения файлов;
- части отказов связи.

Система пока не претендует на уровень высокоанонимного мессенджера. Ее цель — устойчивая локальная связь в ограниченной сети с приемлемой безопасностью, а не абсолютная неотслеживаемость метаданных.
