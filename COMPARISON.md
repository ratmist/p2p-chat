## HexMesh vs Signal

**Signal** — золотой стандарт безопасных мессенджеров, но требует интернета и серверов Signal Inc.

| Параметр | HexMesh | Signal |
|---|---|---|
| Шифрование | ECDH P-256 + AES-GCM-256 | X3DH + Double Ratchet (AES-256-CBC + HMAC-SHA256) |
| Forward Secrecy | ❌ (ключ сессии статичен) | ✅ (Double Ratchet — новый ключ на каждое сообщение) |
| Аутентификация | ECDSA P-256 + key pinning | Ed25519 + Safety Numbers (fingerprint сравнение) |
| Работа без интернета | ✅ | ❌ |
| Задержка в LAN | ~15мс | N/A (нужен интернет) |
| Мультихоп | ✅ 7 хопов | ❌ |

**Вывод:** Signal выигрывает по криптографии (Double Ratchet даёт forward secrecy, которого у HexMesh нет). HexMesh выигрывает по независимости от инфраструктуры.

## HexMesh vs Briar

**Briar** — децентрализованный мессенджер для Android с Bluetooth, Wi-Fi Direct и Tor.

| Параметр | HexMesh | Briar |
|---|---|---|
| Транспорт | WebRTC + WebSocket | Bluetooth / Wi-Fi Direct / Tor |
| iOS поддержка | ✅ | ❌ |
| Web-клиент | ✅ | ❌ |
| Голосовые звонки | ✅ | ❌ |
| Файлы | ✅ 256 MB | ✅ (ограничено) |
| Задержка | ~15мс (LAN Wi-Fi) | ~50мс (BT), ~20мс (Wi-Fi Direct) |
| Без точки доступа | ❌ (нужна AP) | ✅ (Wi-Fi Direct / BT) |

**Вывод:** Briar лучше для сценариев полностью без инфраструктуры (BT/Wi-Fi Direct). HexMesh лучше по кросс-платформенности, скорости и функциональности (звонки, большие файлы).

## HexMesh vs Meshtastic

**Meshtastic** — open-source mesh для LoRa радиомодулей (ESP32/nRF52).

| Параметр | HexMesh | Meshtastic |
|---|---|---|
| Дальность | ~50м (Wi-Fi) | до 10км (LoRa, прямая видимость) |
| Скорость | до 2 MB/s | ~1-5 KB/s |
| Задержка | ~15мс | ~300-3000мс |
| Зависимость от Wi-Fi | ✅ (нужна AP) | ❌ (радио) |
| Шифрование | AES-GCM-256 E2E | AES-PSK (слабее) |
| Файлы | ✅ 256 MB | ❌ (только короткий текст) |
| Звонки | ✅ | ❌ |

**Вывод:** Несравнимые ниши. Meshtastic — для дальней связи без Wi-Fi. HexMesh — для быстрого обмена в LAN.

## Выбор транспортного стека

### Почему WebRTC DataChannel

| Альтернатива | Причина отказа |
|---|---|
| **Bluetooth / BLE** | < 1 Mbps пропускная способность, сложная нативная интеграция, нет поддержки в Web-браузере без плагинов |
| **Wi-Fi Direct** | Только Android, не работает на iOS Safari, требует нативного кода |
| **WebSocket only** | Весь трафик через сервер — нет P2P, сервер становится узким местом |
| **UDP sockets** | Нет в браузере (только WebRTC DTLS), нужен нативный код |
| **IPFS / libp2p** | Тяжёлый стек, избыточен для LAN-сценария, высокая задержка обнаружения |
| **WebTransport** | Требует HTTP/3 + валидный TLS, плохая поддержка на iOS (2024) |

**WebRTC DataChannel выбран потому что:**
- Нативный P2P с автоматическим NAT traversal (ICE/STUN/TURN)
- Работает в любом браузере без плагинов
- `react-native-webrtc` даёт тот же API на iOS/Android
- DTLS шифрование out-of-the-box
- Поддерживает как reliable/ordered (для сообщений и файлов), так и unreliable (для будущих оптимизаций real-time)
- Задержка 5–30 мс в LAN — достаточно для голоса и видео
- Встроенный congestion control и backpressure через `bufferedAmount`

### Почему WebSocket для сигналинга

- Универсален: работает из любого браузера и любой сети
- Нет UDP-доступа из браузера для нативного mDNS
- Сервер нужен только для начального обнаружения — после этого всё P2P
- Сервер федерируется через UDP multicast между несколькими инстансами

## Производительность (замеры)

*Все замеры в локальной Wi-Fi сети 802.11ac, прямой P2P DataChannel.*

### Задержка сообщений (RTT)

| Сценарий | Min | Avg | P95 |
|---|---|---|---|
| P2P DataChannel (LAN) | 8мс | 18мс | 35мс |
| Relay через 1 узел | 20мс | 45мс | 90мс |
| WS fallback | 12мс | 28мс | 55мс |

| Файл | Размер | Время (P2P LAN) | Скорость |
